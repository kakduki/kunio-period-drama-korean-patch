# plan: kunio-autoloop 웹훅 자동 실행 파이프라인 정비

## 결정해야 할 포인트
1) 웹훅 응답을 자연어 지시에서 실제 실행 가능한 자동화로 바꿔야 하는가
- 선택: 변경
- 이유: 현재 `kunio-autoloop` 프롬프트는 텍스트 지시 중심이라 에이전트 판단 편차가 크고, 최근 로그에서 실행이 누락/차단되어 실제 처리가 안 된 이력이 있음.

2) 스크립트를 어디에 둬서 반복 동작을 고정할지
- 선택: 기존 `scripts/kunio_webhook_autorun.py`를 주 실행기로 사용
- 이유: 경로/기능이 이미 레포지토리 내부에 존재해 후속 호출이 단순하고, 향후 유지보수 지점이 1곳으로 집중됨.

3) 목표1/2/3을 어떤 명령으로 정의할지
- 선택: 목표1=텍스트 탐지류 명령, 목표2=폰트/CHR 분석/매핑류 명령, 목표3=IPS 패치 생성류 명령으로 고정
- 이유: 현재 리포지토리에 해당 단계에 대응되는 스크립트 조합이 있어 추가 의존성이 낮고, 실패·성공 로그 수집이 쉬움.

4) Claude CLI 사용 정책
- 선택: 로컬 단계 실패 시 1회 전용 폴백으로 `claude -p` 호출, 인증/결과가 없으면 즉시 원인 로그 후 종료
- 이유: 요청 조건("가능한 경우")을 만족하면서, 인증 실패 같은 실질 실패 요인을 분명히 드러내기 위함.

5) 무한 루프 방지
- 선택: `루프 종료` 또는 `DONE` 텍스트 감지 시 즉시 중단, 그리고 최대 반복 회차 제한 도입
- 이유: 사용자 지정 중단 조건을 정확히 반영하고, 비정상적으로 장시간 반복되는 것을 차단.

## 구현 방식
- `scripts/kunio_webhook_autorun.py`를 실행형 파이프라인으로 재작성.
- 실행 순서: 300초 대기 → repo/ref 검증 → git pull → 상태 출력 → 목표1/2/3 순차 실행 → 변경사항 감지 → 변경 있으면 commit/push.
- 최종 응답을 정확히 `클로드 웹훅 감지 -> HEAD=<sha> -> 실행결과=<요약>` 1줄로 출력.
